## Path 3 · MCP server (`apps/mcp-server`)

`apps/mcp-server/src/server.ts` registers 5 MCP tools per the MCP spec:
1. `ivaronix_ask` — run an Ivaronix skill against text input · optionally anchor a receipt
2. `ivaronix_verify_receipt` — resolve a receipt by on-chain numeric id or 0x bytes32 receiptRoot and report state
3. `ivaronix_search_memory` — local memory engine 4-way hybrid (vector + FTS + temporal + KV)
4. `ivaronix_install_skill` — list available skills, optionally filtered
5. `ivaronix_passport_show` — read on-chain AgentPassport profile for a wallet address

Server registration name: `ivaronix-mcp v0.0.1`. The server runs over stdio per MCP spec; an external Claude Desktop / Cursor / external MCP client can spawn it via `npx ivaronix-mcp` (post-publish) and invoke any of the 5 tools.

The `ivaronix_verify_receipt` tool's implementation calls into `@ivaronix/og-chain`'s V2-first reader (per `apps/mcp-server/src/server.ts:e300781` task #295 closure). Cross-machine smoke: an external client connects via stdio, issues `tools/call ivaronix_verify_receipt {id: 78}`, the server returns the same chain-state confirmed by Path 2.

Honest finding: full stdio JSON-RPC roundtrip from an external MCP client isn't captured in this fire because it requires an external client (Claude Desktop / Cursor) that's operator-driven.
